# 2025-09-25 이미지 저장 방식(DB vs 파일) 검토

## 배경
기존 게시판에서 게시글을 작성할 때 운영자가 이미지를 첨부파일로 업로드하는 구조였음. 이 게시판은 본사 사이트의 컨텐츠를 그대로 복사하여 제공하는 서비스였음. 
반복적인 작업으로 본사에게 데이터 요청을 하였고, 본사는 월 DB 파일을 주게 되었음.

## 문제
본사 사이트의 이미지는 SQL Sever의 Table에 Column에 바이너리 데이터로 이미지를 관리하고 있었음.

본사 DB를 그대로 활용한다면, 기존 업로드 된 게시글 모두 DB 바이너리 컬럼으로 저장해야함.
그럴 경우:
- DB 용량 증가
- 조회/관리 부담 증가
- 기존 첨부파일에 대한 확인 작업 필요
- 이미지파일 -> 바이너리 데이터 검증 작업 필요

본사 데이터 구조를 따라갈 것인가? 현재 운영 시스템의 구조를 유지할 것인가?

## 현재 결론
현재 운영 시스템의 구조를 유지하는 방향을 선택하였다.
구현 방식은 다음과 같다.

1. 본사에서 제공한 SQL Server 바이너리 데이터를 로컬 환경에서 이미지 파일로 변환
2. 변환된 이미지를 Web API를 통해 서버에 업로드
3. 기존 게시판의 첨부파일 관리 방식을 그대로 활용

선택한 이유는 다음과 같다.

### 기존 운영 구조를 크게 변경하고 싶지 않았음

기존 게시판은 첨부파일 기반으로 운영되고 있었으며, 운영자도 해당 방식에 익숙하였다.

본사 DB 구조를 그대로 따를 경우 이미지 관리 방식 자체를 변경해야 했고, 기존에 운영 중인 구조와 관리 방식까지 함께 수정해야 했다.

### 향후 데이터 제공 방식 변경 가능성을 고려하였음

당시에는 SQL Server DB 파일을 전달받고 있었지만, 담당자 변경이나 시스템 개편으로 인해 향후 API 등 다른 형태의 데이터 제공 방식이 추가될 가능성도 있다고 생각하였다.

특정 DB 구조에 강하게 의존하는 구조는 장기적으로 유지보수에 불리할 수 있다고 판단하였다.

### 운영 환경의 부담을 최소화하고 싶었음

이미지 바이너리를 DB에 저장할 경우 데이터베이스 용량 증가와 관리 부담이 발생한다.

반면 이미지 변환 작업은 로컬 환경에서 일회성으로 수행할 수 있으며, 운영 서버에는 최소한의 영향만 주는 구조였다.

### 구현 난이도와 일정도 고려하였음

콘솔 애플리케이션과 Web API만으로 필요한 기능을 구현할 수 있었다.

새로운 저장 구조를 설계하거나 기존 게시판의 저장 방식을 변경하는 것보다 위험이 적고 빠르게 적용할 수 있는 방법이라고 판단하였다.

### 원본 이미지 자체에 대한 의문이 있었음

제공된 이미지들은 웹 서비스에 최적화된 형태라고 보기 어려웠다.

따라서 이미지를 DB에 그대로 저장하는 방식이 반드시 효율적인 구조라고 생각되지는 않았다.

### 외부 API 이미지 경로 리소스 제공

현재 API는 이미지 바이너리 데이터가 아니라 이미지 리소스 경로를 제공하는 구조였다.

이미지를 DB 바이너리로 저장할 경우, 기존처럼 단순히 이미지 경로를 내려주는 방식으로 처리하기 어렵다.  
이 경우 이미지 조회 전용 API를 별도로 만들거나, API 응답에 바이너리 또는 Base64 데이터를 포함하는 방식으로 변경해야 한다.

이는 기존 프론트 표시 방식과 API 응답 구조까지 함께 변경해야 하는 작업이므로, 단순한 저장 방식 변경 이상의 영향이 발생한다.

따라서 기존 API가 이미지 경로를 제공하는 구조를 유지하려면, 이미지를 파일로 변환하여 저장하고 DB에는 경로 정보를 관리하는 방식이 더 자연스러웠다.

프론트 표시 방식에서 `data:image/jpeg;base64, ` 정보를 같이 포함해야한다. 본사 원본 이미지에서는 다양한 이미지 확장자를 사용하기 때문에, MIME 타입도 같이 응답해야한다.
