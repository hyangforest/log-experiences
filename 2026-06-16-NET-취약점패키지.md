# 2026-06-16 AutoMapper 제거 및 의존성 재검토

## 배경

ASP.NET Core / .NET 7.0 기반 프로젝트에서 요청 DTO를 임시 저장용 엔티티로 변환하는 과정이 존재하였다.

두 객체는 약 46개의 동일한 필드를 가지고 있어, 과거에는 반복적인 프로퍼티 대입 코드를 줄이기 위해 `AutoMapper` 기반의 공통 매핑 Helper가 도입되었다.

Helper는 전역으로 구현되어 있었으며 다음과 같이 사용하고 있었다.

```csharp
var tempData = request.MapTo<TDestination>();
```

최근 패키지 취약점 경고가 발생하면서 해당 의존성을 계속 유지할지 검토하게 되었다.

## 문제

취약점 대응을 위해 AutoMapper 패키지를 업데이트할지, 아니면 의존성을 제거할지 판단해야 했다.

## 고민

* 요청 DTO와 엔티티 사이에는 약 46개의 동일한 필드가 존재한다.
* 당시에는 수동 매핑 코드를 작성하는 부담을 줄이기 위해 AutoMapper를 도입한 것으로 보인다.
* 하지만 공통 매핑 Helper는 전역으로 구현되어 있었음에도 약 3년 동안 사용처가 단일 컨트롤러의 특정 엔드포인트 1개뿐이었다.
* 프로젝트 전반에서 공통 매핑 라이브러리로 활용되거나 사용 범위가 확장되지 않았다.
* 매핑 대상 역시 여러 기능에서 공용으로 사용하는 테이블이 아니라 특정 기능에서만 사용하는 임시 저장용 테이블이었다.
* 단순 필드 복사를 위해 취약점 관리 대상인 외부 패키지를 계속 유지하는 것이 적절한지 고민하게 되었다.

## 현재 결론

과거에는 46개의 필드를 반복해서 대입하지 않기 위한 합리적인 선택이었다고 판단하였다.

하지만 현재 시점에서는 실제 사용 범위가 매우 제한적이고, 단순 필드 복사 외의 복잡한 매핑 기능도 사용하지 않고 있었다. 또한 매핑 대상이 특정 기능에서만 사용하는 임시 저장용 테이블이라는 점까지 고려하면, 공통 매핑 라이브러리를 유지할 필요성이 크지 않다고 판단하였다.

따라서 직접 매핑 코드로 변경하고 AutoMapper 의존성을 제거하는 방향을 선택하였다.

## 의문

* 동일한 필드가 40~50개 이상인 DTO-엔티티 쌍이 여러 개 존재한다면, 여전히 직접 매핑이 더 적절할까?
* 여러 컨트롤러나 서비스에서 동일한 매핑 로직을 반복해서 사용했다면, 직접 매핑을 유지했을까, 아니면 공통 매핑 메서드나 매핑 라이브러리를 선택했을까?
* 라이브러리 도입과 제거는 현재의 사용 범위만 판단해야 할까, 아니면 미래의 확장 가능성까지 고려해야 할까?
* 과거에는 합리적이었던 기술 선택이 현재에는 기술 부채가 되는 시점을 어떻게 판단할 수 있을까?
