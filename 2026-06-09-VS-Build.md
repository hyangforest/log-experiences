# Visual Studio 2022에서 ASP.NET Web Forms Web Site 빌드 실패

## 배경

코드 형상관리 플랫폼을 Azure DevOps에서 GitHub로 이관하는 작업을 진행하고 있다.

현재 운영 중인 프로젝트 중에는 ASP.NET Web Forms Web Site 형식의 프로젝트가 존재한다.

.NET Framework 4.5 기반 프로젝트는 Visual Studio 2022에서 정상적으로 개발 및 빌드가 가능했지만, 
.NET Framework 3.5 기반 프로젝트는 웹사이트 빌드 시 참조 중인 다른 프로젝트의 네임스페이스를 찾지 못하는 오류가 발생하였다.

과거 GitLab에서 Azure DevOps로 형상관리 플랫폼을 이관할 때도 동일한 현상을 경험한 적이 있다.

당시 부장님은 "Visual Studio 2022의 Core와 Common이 겹치는 것 같다"는 취지로 이야기하셨고, 나 역시 특별히 깊게 의심하지 않은 채 그 설명을 받아들이고 Visual Studio 2019를 사용하며 넘어갔다.

하지만 이번 GitHub 이관 과정에서 다시 같은 문제가 발생하면서, 당시 추측했던 원인이 실제 원인이었는지 다시 확인해보고 싶어졌다.

---

## 문제

Visual Studio 2022에서 Web Site 빌드를 수행하면 다음과 같은 오류가 발생하였다.

* `'Common' 형식 또는 네임스페이스 이름을 찾을 수 없습니다.`
* `'DbHelperParam' 형식 또는 네임스페이스 이름을 찾을 수 없습니다.`

반면 동일한 소스는 Visual Studio 2019에서 정상적으로 빌드되었다.

---

## 확인한 내용

### Common 프로젝트

* `Common.dll` 정상 생성
* 빌드 결과물이 `Web/Bin/Common.dll`로 정상 복사됨
* `DbHelperParam` 클래스도 실제 존재함

### Web Site 프로젝트

* 참조 목록에 `Common` 존재
* `Web/Bin/Common.dll` 존재
* Web Site 형식 프로젝트 사용

### 기타 확인

* NuGet 패키지 누락 여부 확인
* DLL 복사 실패 여부 확인
* 파일 잠금 여부 확인
* 참조 경로 이상 여부 확인

모두 특별한 문제는 발견되지 않았다.

### Framework 버전 확인

* Target Framework : `.NET Framework 3.5`
* Visual Studio 2022 : `.NET Framework 3.5 개발 도구` 설치 가능
* 그러나 `.NET Framework 3.5 Targeting Pack`은 제공되지 않음

MSDN 문서에서도 Visual Studio 2022는 오래된 .NET Framework 버전에 대한 지원이 축소되었음을 확인할 수 있었다.

[참고](https://learn.microsoft.com/ko-kr/dotnet/framework/install/guide-for-developers)

> Visual Studio 2022 이상 버전은 .NET Framework 4.0 ~ 4.5.1을 대상으로 하는 앱을 빌드할 수 없습니다.
>
> 이러한 앱을 계속 빌드하려면 Visual Studio 2019 또는 이전 버전을 사용해야 합니다.

---

## 고민

처음에는 아래와 같은 기술적 문제를 의심했다.

* Common.dll 참조 누락
* Common 프로젝트의 NuGet 패키지 복원 실패
* Common 프로젝트 빌드 출력 Web/Bin 복사 실패
* 참조 경로 문제
* Visual Studio 캐시 문제

하지만 다음 사실이 계속 걸렸다.

* VS 2019에서는 정상 동작
* VS 2022에서만 동일 증상 발생
* Common.dll은 실제로 정상 생성됨

즉, 프로젝트 자체의 문제라기보다는 개발 환경 또는 개발 도구의 차이일 가능성이 높다고 판단하였다.

조사 과정에서 프로젝트가 매우 오래된 Web Site 형식이며, 참조하는 클래스 라이브러리 Common 프로젝트는 Target Framework가 .NET Framework 3.5라는 점을 확인하였다.

그 이후부터는 개별 DLL 문제가 아니라 Visual Studio 2022의 지원 범위 문제라는 가설이 더 설득력 있게 느껴졌다.

---

## 현재 결론

이번 문제의 핵심은 Common 프로젝트가 아니라 개발 환경의 호환성 문제로 판단하였다.

Visual Studio 2022는 최신 .NET 및 ASP.NET Core 중심으로 발전하면서 오래된 .NET Framework 및 Web Site 형식 프로젝트에 대한 지원 우선순위가 크게 낮아졌다.

특히 .NET Framework 3.5 환경은 Visual Studio 2022에서 완전한 개발 대상으로 고려되지 않는 것으로 보인다.

처음에는 Common.dll 참조, NuGet 패키지, Web/Bin 복사 문제 등을 의심했지만, 동일한 프로젝트가 Visual Studio 2019에서는 정상적으로 동작한다는 점이 중요한 단서가 되었다.

또한 과거 GitLab → Azure DevOps 이관 당시에도 동일한 문제가 있었지만, 당시에는 "Visual Studio 2022의 Core와 Common이 겹치는 것 같다"는 설명을 받아들이고 Visual Studio 2019를 사용하는 것으로 정리했었다.

이번 GitHub 이관 과정에서 동일한 문제가 다시 발생하면서 원인을 추적해본 결과, 특정 DLL이나 프로젝트 설정의 문제가 아니라 .NET Framework 3.5, Web Site 형식, Visual Studio 2022 지원 범위가 결합된 호환성 문제일 가능성이 높다고 판단하였다.

결과적으로 이번 경험을 통해 프로젝트 자체의 문제와 개발 도구의 지원 범위 문제를 구분해서 볼 필요가 있다는 점을 확인할 수 있었다.

따라서 현재 시점에서는 Visual Studio 2022에 억지로 맞추기보다 Visual Studio 2019를 기준 개발 환경으로 유지하는 것이 가장 현실적이고 안전한 선택으로 보인다.

장기적으로는 Web Site → Web Application 전환, .NET Framework 버전 상향 여부를 검토할 필요가 있다.

## 의문

### 기술적인 의문

* Web Site 형식이 아닌 Web Application이었다면 VS 2022에서도 정상 빌드되었을까?
* 실제로 문제가 되는 것은 .NET Framework 3.5인가, Web Site 형식인가?
* .NET Framework 4.0~4.5.1과 .NET Framework 3.5 지원 축소 배경은 무엇일까?

### 운영 관점의 의문

* 이 프로젝트를 앞으로도 VS 2019에 묶어두는 것이 맞을까?
* 장기적으로 Web Application 전환이 필요할까?
* .NET Framework 3.5를 유지해야 하는 실질적인 이유가 남아 있을까?

### Microsoft 관점의 의문

* Visual Studio 2022의 64비트 전환과 관련된 기술적 제약이 있었을까?
