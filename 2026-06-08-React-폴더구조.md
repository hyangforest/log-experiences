# 2026-06-08 React 폴더구조

## 배경

React 프로젝트의 컴포넌트 관리 구조에 대한 고민이 생김.
Kiro의 Testing Strategy를 검토하면서 함수 분리, 컴포넌트 분리, 테스트 용이성을 위한 구조 변경에 대해 생각하게 되었고, 페이지 전용 컴포넌트를 어디에 두는 것이 적절한지 고민함.

## 문제
페이지 전용 컴포넌트와 공통 컴포넌트의 관리 위치를 어떻게 구분할 것인가. React 프로젝트에서 폴더 구조를 어떤 기준으로 구성하는 것이 유지보수에 유리한가.

## 고민
React 프로젝트에서 페이지 전용 컴포넌트를 어디에 둘 것인지도 고민 함.
- 컴포넌트 모음 폴더 안에 페이지별 컴포넌트를 관리하는 것이 직관적인가?
- 페이지 하위에 `components` 폴더를 두는 것이 일반적인가?
- 페이지 전용 컴포넌트와 공통 컴포넌트는 어떤 기준으로 구분해야 하는가?

예시:
- components/setting
- pages/setting/components
- pages/setting/menu1/components


## 현재 결론
- 공통 UI는 `components` 아래에 둔다.
- 페이지 전용 페이지 또는 메뉴에서만 사용하는 컴포넌트는 해당 페이지에 `components` 폴더를 만들어 하위에 둔다.
  - `componets` 폴더가 계속 생길 경우에 검색 결과, 위치, git commit 할 때 위치 확인 등 많이 생겨버리면 헷갈렸음. 하지만 페이지와 관련 컴포넌트가 가까이 위치할수록 해당 컴포넌트의 사용 범위를 파악하기 쉽고, 유지보수 시에도 관련 파일을 함께 찾을 수 있어 탐색 비용이 줄어든다고 판단함.
  - 이미 `src'의 `components`는 사용자의 공통 UI 컴포넌트를 사용하고 있었고, 그 아래 `admin`의 `components` 폴더는 관리자용 페이지의 공통 UI 컴포넌트를 사용하고 있었음. 따라서 페이지 단위 컴포넌트도 동일한 방식으로 관리하는 것이 일관성 측면에서 자연스럽다고 판단함.

## 의문
1. 왜 사용자 사이트와 관리자 사이트는 같은 프로젝트에서 개발을 할까? 
  - 서브도메인으로 분리했을 때의 장단점은 무엇일까?
  - 사용자 사이트와 관리자 사이트는 스타일과 목적이 많이 다른데, 하나의 프로젝트로 운영하는 이유는 무엇일까?
  - 인증, 배포, 운영 비용 측면에서 이점이 있는 것일까?
2. 전형적인 React 폴더 구조라는 것이 존재할까?
  - React는 자유도가 높은 만큼 정답이 없는 것일까?
  - 규모에 따라 권장되는 구조가 달라지는 것일까?
  - 실제 현업에서는 어떤 기준으로 구조를 선택하는 것일까?
