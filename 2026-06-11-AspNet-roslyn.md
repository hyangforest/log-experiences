# 2026-06-11 Roslyn(csc.exe) 경로 오류

## 배경

과거 ASP.NET Web Forms 프로젝트들을 아카이빙용 Git 저장소로 정리하는 과정에서 빌드 검증을 수행하였다.

프로젝트마다 개발 시점과 개발 환경이 달라 현재 환경에서 정상 빌드되지 않는 사례가 일부 발견되었다.

## 문제

빌드 시 아래와 같은 오류가 발생하였다.

```text
'D:\...\bin\roslyn\csc.exe' 경로의 일부를 찾을 수 없습니다.
```

동일한 문제가 서로 다른 프로젝트에서 2건 발견되었다.

## 고민

처음에는 NuGet 패키지 복원 누락이나 DLL 누락과 같은 일반적인 참조 문제를 의심하였다.

오류 메시지에는 `bin\roslyn\csc.exe` 경로를 찾을 수 없다고 표시되었지만, Roslyn이 어떤 역할을 하는지 알지 못해 단순히 누락된 파일 정도로 생각하였다.

해결 과정에서 Visual Studio 기본 템플릿을 통해 추가된 Roslyn CodeDom Provider 설정이 존재한다는 점을 확인하였다.

현재 프로젝트는 ASP.NET Web Site 방식으로 구성되어 있었으며, Roslyn을 사용하려면 관련 파일이 `bin\roslyn` 폴더에 함께 존재해야 한다.

하지만 프로젝트에는 해당 파일이 존재하지 않았고, Git 저장소에도 포함되어 있지 않았다.

또한 ASP.NET Web Site 방식은 프로젝트를 처음 열었을 때 NuGet 패키지가 자동 복원되지 않는다.

실제로 패키지 복원을 수행하자 Roslyn 관련 파일도 함께 다운로드되었으며, 처음에는 이것이 문제 해결 방법처럼 보였다.

하지만 복원 과정에서 현재 프로젝트에서 사용하지 않는 여러 패키지까지 함께 설치되는 것을 확인하였다.

결과적으로 Roslyn 관련 설정이 남아 있었기 때문에 불필요한 패키지까지 계속 복원되고 있었으며, 실제 문제는 누락된 파일보다 프로젝트에 사용하지 않는 설정과 패키지가 정리되지 않은 상태에 있었다고 판단하였다.

## 현재 결론

문제의 원인은 Roslyn 기능 자체가 아니라 프로젝트에 남아 있던 Roslyn 관련 설정과 의존성 관리 상태에 있었다.

Roslyn은 ASP.NET에서 최신 C# 컴파일러를 사용하기 위해 추가된 기능이지만, 현재 프로젝트에서는 해당 기능을 사용하지 않고 있었다.

과거 Roslyn 관련 파일은 제거되었으나 `web.config`의 CodeDom 설정과 `packages.config`의 패키지 정보는 그대로 유지되고 있었다.

그 결과 프로젝트는 Roslyn 컴파일러를 찾도록 설정되어 있었고, 신규 개발 환경에서는 불필요한 패키지 복원이 발생하거나 `bin\roslyn\csc.exe` 관련 오류가 발생할 수 있는 상태였다.

Roslyn 관련 설정과 패키지를 제거한 후에도 정상적으로 빌드되는 것을 확인하였으며, 사용하지 않는 기능에 대한 설정과 의존성을 함께 정리하는 것이 적절한 해결 방법이라고 판단하였다.

## 의문
- Microsoft는 왜 실제로 사용하지 않는 기능까지 기본 템플릿에 포함하도록 설계했을까?
- 개발 편의성을 위한 선택이었을까, 아니면 최신 기능 사용을 유도하기 위한 전략이었을까?
- 기본 템플릿에 포함된 기능들은 시간이 지나도 계속 유지해야 할까, 아니면 프로젝트 상황에 맞게 정리해야 할까?
- 사용하지 않는 패키지와 설정은 어느 시점에 정리하는 것이 적절할까?
- 현재 운영 중인 다른 레거시 프로젝트에도 비슷하게 사용하지 않는 설정이 남아 있을까?
